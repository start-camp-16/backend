# 추천 랭킹·전체 장소 API 분리 설계

## 목적

기존 `/api/rankings`가 원본 배열 순서를 랭킹처럼 반환하는 문제를 없애고, 시연용 추천 목록과 전체 장소 탐색 목록의 책임을 분리한다.

- `/api/rankings`: `구 × 카테고리`별 추천 장소 최대 5개
- `/api/locations`: 추천 여부와 무관한 전체 장소 페이지네이션 목록

추천은 실제 인기나 평점을 의미하지 않는다. 현재 원본 데이터와 가벼운 편집 기준으로 만든 시연용 추천 순서다.

## 범위

### 포함

- 현재 적재된 7개 장소 카테고리와 6,518개 장소
- 원본 데이터가 존재하는 `구 × 카테고리` 조합별 최대 5개 추천
- 추천 데이터의 정적 파일 관리
- 두 조회 API와 공유 OpenAPI 계약 변경
- 백엔드 통합·계약 테스트
- 프론트엔드 전달용 구현 프롬프트

### 제외

- 외부 리뷰, 검색량, 클릭, 저장 데이터 수집
- 실시간 인기 점수와 개인화
- 음식점 카테고리 추가
- 원본에 장소가 없는 조합을 인접 지역 장소로 채우는 처리
- 축제의 현재 개최 여부 판정

## 추천 데이터

원본 JSON은 라이선스 조건에 따라 수정하지 않는다. 추천 순서는 별도 `data/recommendations.json`에 저장한다.

파일 최상위는 `recommendations` 배열이며 각 항목은 다음 정보를 가진다.

```json
{
  "recommendations": [
    {
      "district": "강남구",
      "category": "관광지",
      "contentIds": ["123", "456", "789"]
    }
  ]
}
```

규칙:

- 한 조합에는 0~5개의 `content_id`만 둔다.
- 같은 조합과 같은 `content_id`를 중복하지 않는다.
- 모든 ID는 `locations` 테이블에 존재해야 한다.
- ID가 가리키는 장소의 구와 카테고리는 항목의 조합과 일치해야 한다.
- 원본 장소가 없는 조합은 추천 파일에 만들지 않는다.
- 원본 장소가 있는 조합에는 `min(5, 해당 조합의 장소 수)`개의 ID를 둔다.
- 주소가 없는 여행코스는 현재 적재 결과대로 `기타 × 여행코스`에서만 추천한다.

## 선정 기준

시연용 초기 추천은 raw 데이터에서 조합별 후보를 추출하고 다음 기준으로 최대 5개를 선정해 정적 파일로 확정한다.

1. 관광·문화·체험·숙박·쇼핑 등 해당 카테고리의 방문 목적이 제목에서 분명한 장소
2. 공원, 박물관, 미술관, 공연장, 전통시장, 특화거리처럼 고유 방문 목적이 있는 장소
3. 대표 이미지와 주소가 있는 장소
4. 같은 브랜드나 유사 시설이 한 조합을 반복해서 차지하지 않도록 다양화
5. 일반 판매점과 반복 체인점은 후순위
6. 판단 기준이 같으면 기존 `source_order`, `id` 순서

이 기준은 런타임 점수로 노출하지 않는다. 추천의 결과만 `recommendations.json`에 저장하며 이후 파일을 직접 수정해 큐레이션할 수 있다.

## API 설계

### `GET /api/rankings`

필수 쿼리:

- `district`: 빈 문자열이 아닌 구 이름
- `category`: `LocationCategory` 열거형

제거되는 쿼리:

- `page`
- `size`

응답:

```json
{
  "district": "강남구",
  "category": "관광지",
  "items": [
    {
      "rank": 1,
      "content_id": "123",
      "category": "관광지",
      "title": "장소명",
      "address": "서울특별시 강남구 ...",
      "district": "강남구",
      "longitude": 127.0,
      "latitude": 37.5,
      "image_url": "https://...",
      "thumbnail_url": "https://...",
      "phone": null
    }
  ]
}
```

- 추천 파일에 없는 조합은 `200`과 빈 `items`를 반환한다.
- 파일의 `contentIds` 순서가 `rank` 1~5가 된다.
- 응답에서 내부 적재 순서인 `source_order`는 제거한다.

### `GET /api/locations`

선택 쿼리:

- `district`: 지정 시 해당 구만 조회
- `category`: 지정 시 해당 카테고리만 조회
- `page`: 기본 1, 최솟값 1
- `size`: 기본 20, 1~100

응답은 장소 `items`와 기존 `Pagination`을 반환한다. 장소 항목에는 `rank`와 `source_order`를 포함하지 않는다.

정렬:

- 카테고리가 지정되면 `source_order ASC, id ASC`
- 카테고리가 없으면 `category ASC, source_order ASC, id ASC`

## 내부 구성

- `app/locations/recommendations.py`: 추천 파일 로드와 형식 검증
- `app/locations/service.py`: 추천 조회와 전체 목록 조회
- `app/locations/schemas.py`: 공통 장소, 추천 장소, 두 목록 응답 스키마
- `app/locations/router.py`: 두 엔드포인트의 쿼리 선언
- `data/recommendations.json`: 확정된 추천 순서
- `shared/openapi.yaml`: 프론트엔드와 공유하는 기준 계약

추천 파일은 매 요청마다 다시 읽지 않고 프로세스에서 한 번 읽어 불변 인덱스로 사용한다. 서비스는 추천 ID를 DB에서 조회한 뒤 정적 파일의 순서로 재정렬한다.

## 오류와 데이터 불일치

- 잘못된 카테고리, 페이지, 크기는 기존 공통 `400 VALIDATION_ERROR` 형식을 사용한다.
- 존재하지 않는 구 이름은 오류 대신 빈 결과를 반환한다.
- 추천 파일의 형식 오류, 중복, 5개 초과는 애플리케이션이 파일을 처음 불러올 때 실패시킨다.
- 추천 ID가 DB에 없거나 구·카테고리가 일치하지 않는 문제는 데이터 무결성 통합 테스트가 실패시킨다.
- 런타임에서 일부 추천 ID를 찾지 못하면 찾은 항목만 순서를 유지해 반환하고 서버 로그에 경고를 남긴다.

## 테스트

- 추천 API가 파일 순서대로 최대 5개와 연속된 `rank`를 반환한다.
- 추천이 없는 조합과 존재하지 않는 구가 빈 목록을 반환한다.
- 추천 API가 `page`, `size` 없이 동작한다.
- 전체 목록 API가 무필터, 구 필터, 카테고리 필터, 복합 필터를 지원한다.
- 전체 목록의 페이지 순위가 아니라 장소 목록과 페이지 정보만 반환한다.
- 추천 파일의 중복, 5개 초과, 존재하지 않는 ID, 조합 불일치를 검출한다.
- `shared/openapi.yaml`과 FastAPI 응답 모델의 계약 테스트를 갱신한다.

## 호환성

`/api/rankings`의 페이지네이션 제거와 응답 구조 변경은 기존 프론트엔드에 대한 의도적인 계약 변경이다. 프론트엔드는 `docs/frontend-ranking-api-prompt.md`에 따라 랭킹과 전체 목록을 분리해 연동해야 한다.
