# 규칙 기반 코스 생성·익명 공유 설계

## 목적

사용자가 서울의 구, 관심 카테고리, 방문 장소 수를 선택하면 이동 동선이 짧은 코스 초안을 생성하고, 사용자가 장소와 순서를 편집한 뒤 익명으로 저장·공유할 수 있게 한다.

이 기능은 실제 도로 경로나 이동시간을 제공하지 않는다. 현재 보유한 장소 좌표를 이용한 직선거리 기반 추천이며, 같은 입력에는 항상 같은 결과를 반환한다.

## 범위

### 포함

- 구, 관심 카테고리 1~3개, 장소 수 3~5개를 이용한 코스 초안 생성
- 기존 큐레이션 랭킹과 장소 데이터를 이용한 결정적 추천
- 코스 장소 추가·삭제·순서 변경 후 최종 구성 저장
- 공개 링크를 통한 비회원 조회
- 비밀번호 확인을 통한 코스 수정·삭제
- 코스별 장소 3~5개 및 중복 방지 검증
- 공유 OpenAPI 계약과 자동화 테스트 갱신

### 제외

- 회원, 세션, 사용자별 코스 목록
- 실제 도로 길찾기, 대중교통·도보 경로, 예상 이동시간
- 장소별 체류 시간과 운영시간을 고려한 일정표
- AI를 이용한 장소 선정 또는 코스 설명 생성
- 좋아요, 조회수, 검색 노출용 공개 코스 목록
- 코스 공동 편집과 수정 이력

## 사용자 흐름

1. 사용자가 구, 관심 카테고리, 방문 장소 수를 선택한다.
2. 서버가 조건에 맞는 장소를 선정하고 직선거리가 짧아지도록 방문 순서를 정한 임시 코스를 반환한다.
3. 사용자가 프론트엔드에서 장소를 추가·삭제하거나 순서를 변경한다.
4. 사용자가 제목과 수정·삭제용 비밀번호를 입력해 최종 코스를 저장한다.
5. 서버가 예측하기 어려운 공개 ID를 발급한다.
6. 누구나 공개 ID가 포함된 링크로 코스를 조회할 수 있다.
7. 수정과 삭제 요청에만 비밀번호를 요구한다.

자동 추천 결과는 서버에 저장하지 않는다. 사용자가 확정한 코스만 저장해 반복 추천으로 불필요한 데이터가 쌓이지 않게 한다.

## 데이터 모델

### `courses`

| 필드 | 타입 | 제약/설명 |
|---|---|---|
| `id` | integer | PK |
| `public_id` | string | unique, URL 공유용 예측 불가능 식별자 |
| `title` | string | trim 후 1~100자 |
| `password` | string | 4~20자, 응답·로그 제외 |
| `created_at` | datetime | UTC 저장 |
| `updated_at` | datetime | UTC 저장 |

`public_id`는 UUID4의 하이픈 없는 32자 문자열로 생성한다. 클라이언트가 값을 지정할 수 없으며 충돌 시 서버가 다시 생성한다.

비밀번호 저장과 비교는 기존 익명 게시글·댓글의 MVP 정책을 따른다. 비밀번호는 API 응답 스키마에 선언하지 않고 요청 로그에도 기록하지 않는다.

### `course_stops`

| 필드 | 타입 | 제약/설명 |
|---|---|---|
| `id` | integer | PK |
| `course_id` | integer | FK → `courses.id`, cascade delete |
| `location_id` | integer | FK → `locations.id` |
| `position` | integer | 1~5의 방문 순서 |

무결성 제약:

- `(course_id, position)`은 유일하다.
- `(course_id, location_id)`는 유일하다.
- 한 코스는 정확히 3~5개의 방문지를 가진다. 이 개수는 서비스 계층의 트랜잭션 안에서 검증한다.
- 장소 정보는 복사하지 않고 기존 `locations`를 참조한다.

코스를 조회할 때 현재 장소 정보를 결합한다. 구간별 직선거리는 저장하지 않고 현재 좌표를 기준으로 계산한다.

## 자동 추천 알고리즘

### 입력 검증

- `district`: 공백 제거 후 1자 이상
- `categories`: 서로 다른 장소 카테고리 1~3개
- `stop_count`: 3~5
- 카테고리 개수는 `stop_count` 이하여야 한다.

### 후보 구성

1. 선택한 구와 카테고리에 속한 장소를 조회한다.
2. 경도·위도가 null이거나 `0, 0`인 장소를 제외한다.
3. 각 카테고리의 기존 `data/recommendations.json` 추천 장소를 먼저 배치한다.
4. 추천 장소가 부족하면 대표 이미지와 주소가 모두 있는 장소, 둘 중 하나만 있는 장소, 둘 다 없는 장소 순으로 보충한다.
5. 같은 우선순위에서는 `source_order ASC, id ASC`를 사용한다.

이 정렬은 데이터가 바뀌지 않는 한 같은 입력이 같은 후보 순서를 갖게 한다.

### 장소 선정과 순서 결정

1. 첫 방문지는 전체 후보 중 추천 우선순위, 카테고리 입력 순서, `source_order`, `id` 순으로 결정한다.
2. 다음 방문지를 고를 때 현재까지 선택된 장소 수가 가장 적은 카테고리를 우선 대상으로 삼는다. 아직 한 번도 포함되지 않은 카테고리가 있으면 그 카테고리가 먼저 대상이 된다.
3. 우선 대상 카테고리가 여럿이면 해당 카테고리들의 미방문 후보 중 현재 방문지와 직선거리가 가장 가까운 장소를 선택한다.
4. 거리까지 같으면 후보 구성의 결정적 정렬 순서를 사용한다.
5. 요청한 `stop_count`에 도달할 때까지 반복한다.

카테고리 입력 배열의 순서는 동일 거리·동일 우선순위의 최종 동률 해소에만 사용한다. 선택 가능한 후보가 있는 한 요청한 모든 카테고리가 최소 한 번 포함되고, 이후 방문지는 카테고리별 개수가 가능한 한 균등해진다.

직선거리는 두 좌표에 Haversine 공식을 적용해 미터 단위 정수로 반올림한다. 첫 방문지의 `distance_from_previous_meters`는 null이고, 이후 방문지는 직전 방문지와의 거리를 가진다. 이 값은 도로 거리나 예상 이동시간으로 표현하지 않는다.

조건에 맞고 좌표가 유효한 서로 다른 장소가 `stop_count`보다 적으면 일부 결과를 반환하지 않고 `400 COURSE_NOT_ENOUGH_LOCATIONS`로 실패한다.

## API 설계

### `POST /api/course-suggestions`

요청:

```json
{
  "district": "종로구",
  "categories": ["관광지", "문화시설"],
  "stop_count": 4
}
```

응답:

```json
{
  "district": "종로구",
  "categories": ["관광지", "문화시설"],
  "stops": [
    {
      "position": 1,
      "distance_from_previous_meters": null,
      "location": {
        "content_id": "123",
        "category": "관광지",
        "title": "장소명",
        "address": "서울특별시 종로구 ...",
        "district": "종로구",
        "longitude": 126.0,
        "latitude": 37.0,
        "image_url": null,
        "thumbnail_url": null,
        "phone": null
      }
    }
  ],
  "total_straight_line_distance_meters": 1200
}
```

추천 결과는 DB에 저장하지 않는다.

### `POST /api/courses`

요청 필드:

- `title`: 1~100자
- `password`: 4~20자
- `location_content_ids`: 방문 순서대로 나열한 서로 다른 장소 ID 3~5개

모든 장소가 존재하는지 확인한 뒤 코스와 방문지를 하나의 트랜잭션으로 저장한다. 성공 시 `201`과 전체 코스 응답을 반환한다.

### `GET /api/courses/{public_id}`

비밀번호 없이 코스를 조회한다. 응답에는 `public_id`, 제목, 생성·수정 시각, 순서가 지정된 장소, 구간별 직선거리, 총 직선거리가 포함된다. 비밀번호는 포함하지 않는다.

저장 코스에는 서로 다른 구나 카테고리의 장소를 함께 둘 수 있다. 자동 추천의 단일 구·선택 카테고리 제한은 초안 생성에만 적용하며, 사용자의 수동 편집을 제한하지 않는다.

좌표가 없는 장소도 사용자가 직접 구성한 저장 코스에는 포함할 수 있다. 두 방문지 중 하나라도 좌표가 없으면 해당 구간 거리를 null로 반환한다. 구간 거리 중 하나라도 null이면 총 직선거리도 null로 반환해 전체 거리를 완전하게 계산할 수 없음을 표시한다.

### `PUT /api/courses/{public_id}`

요청에는 `password`, 새 `title`, 새 `location_content_ids` 전체를 보낸다. 비밀번호와 새 구성을 모두 검증한 뒤 기존 방문지를 교체한다. 검증 또는 저장이 실패하면 기존 코스는 그대로 유지된다. 성공 시 갱신된 전체 코스 응답을 반환한다.

### `DELETE /api/courses/{public_id}`

요청 본문의 `password`를 확인한 뒤 코스와 방문지를 삭제하고 `204`를 반환한다.

## 내부 구성

- `app/courses/models` 역할은 기존 `app/models.py`의 `Course`, `CourseStop` 모델이 담당한다.
- `app/courses/schemas.py`: 추천 요청·응답과 코스 CRUD 요청·응답 모델
- `app/courses/recommender.py`: 후보 정렬, 카테고리 균형 선정, Haversine 거리 계산
- `app/courses/service.py`: 추천 조회와 코스 트랜잭션 처리
- `app/courses/router.py`: `/api/course-suggestions`, `/api/courses` 엔드포인트
- `alembic/versions/0002_add_courses.py`: 코스와 방문지 테이블 마이그레이션
- `shared/openapi.yaml`: 프론트엔드와 공유하는 기준 계약

추천 로직은 DB 조회와 분리된 순수 함수 단위로 구성해, 동일 후보 입력에 대한 선정 결과를 빠르고 결정적으로 테스트할 수 있게 한다.

## 트랜잭션과 오류 처리

코스 생성과 수정은 다음 검증을 모두 통과한 뒤 쓰기를 확정한다.

- 장소 ID가 3~5개인지 확인
- 장소 ID 중복 여부 확인
- 모든 장소가 DB에 존재하는지 확인
- 수정 시 코스 존재 여부와 비밀번호 확인

수정은 기존 방문지를 삭제하고 새 방문지를 추가하는 전체 교체 방식이다. 이 과정 전체를 단일 트랜잭션에서 처리해 중간 상태가 노출되지 않게 한다.

| HTTP | 코드 | 상황 |
|---:|---|---|
| 400 | `VALIDATION_ERROR` | 장소 수, 중복, 제목, 카테고리 등 입력 오류 |
| 400 | `COURSE_NOT_ENOUGH_LOCATIONS` | 자동 추천 후보가 요청한 장소 수보다 적음 |
| 403 | `PASSWORD_MISMATCH` | 수정·삭제 비밀번호 불일치 |
| 404 | `COURSE_NOT_FOUND` | 공개 ID에 해당하는 코스 없음 |
| 404 | `LOCATION_NOT_FOUND` | 저장 구성에 존재하지 않는 장소가 포함됨 |

오류는 기존 공통 `code`, `message`, `details` 구조를 사용한다. 비밀번호, SQL, 내부 예외 메시지는 응답과 로그에 포함하지 않는다.

## 테스트

### 추천 단위 테스트

- 같은 후보와 입력이 항상 같은 장소와 순서를 반환한다.
- 선택 가능한 모든 카테고리가 최소 한 번 포함된다.
- 추가 방문지가 카테고리 사이에 가능한 한 균등하게 배분된다.
- 현재 위치에서 가까운 후보가 선택되고 동률은 결정적 순서로 해소된다.
- Haversine 거리와 미터 단위 반올림을 검증한다.
- 좌표가 null이거나 `0, 0`인 장소를 제외한다.

### API·통합 테스트

- 추천 요청의 구, 카테고리, 카테고리 수, 장소 수를 검증한다.
- 후보 부족 시 일부 목록 대신 `COURSE_NOT_ENOUGH_LOCATIONS`를 반환한다.
- 코스 생성, 공개 조회, 수정, 삭제의 전체 흐름을 검증한다.
- 장소 3개 미만, 5개 초과, 중복, 존재하지 않는 ID를 거부한다.
- 잘못된 비밀번호와 존재하지 않는 공개 ID를 처리한다.
- 수정 실패 시 기존 제목과 방문 순서가 유지된다.
- 코스 삭제 시 방문지가 cascade 삭제된다.
- 좌표 없는 수동 방문지가 포함되면 거리 합계를 null로 반환한다.
- 모든 성공 응답과 오류 응답에 비밀번호가 없는지 검증한다.
- 공유 OpenAPI와 FastAPI 응답 모델의 계약 일치를 검증한다.
- 마이그레이션으로 코스 테이블을 만들 수 있는지 검증한다.

## 완료 조건

- 사용자가 구, 관심 카테고리, 장소 수 3~5개로 결정적인 코스 초안을 받을 수 있다.
- 사용자가 편집한 장소 3~5개를 익명 코스로 저장할 수 있다.
- 공개 ID로 누구나 코스를 조회할 수 있고 비밀번호로만 수정·삭제할 수 있다.
- 추천과 코스 CRUD의 성공·오류 계약이 `shared/openapi.yaml`과 일치한다.
- 자동화 테스트가 추천 결정성, 데이터 무결성, 트랜잭션 안전성, 비밀번호 비노출을 검증한다.
